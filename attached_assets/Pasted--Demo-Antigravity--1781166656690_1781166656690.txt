现在录音功能不稳定，有时候点击麦克风后录不进去，导致最小 Demo 链路没有跑通。请这次只优化“录音/语音输入链路”，不要改动背景、三人格切换、星球、聊天框视觉、Antigravity 动效。

当前目标不是做真实完整 AI，而是先把 Demo 的最小链路稳定跑通：

用户点击麦克风
→ 进入录音中状态
→ 再次点击停止
→ 得到一段文字
→ 自动填入输入框或自动发送
→ 当前选中的人格生成 mock 回复
→ 聊天消息显示出来

一、请先检查当前录音实现

现在录音有时不稳定，请重点检查：

1. 是否正确请求麦克风权限
2. 是否处理浏览器不支持 SpeechRecognition 的情况
3. 是否处理用户拒绝麦克风权限的情况
4. 是否处理录音开始失败的情况
5. 是否处理 recognition.onend 自动触发导致状态混乱的问题
6. 是否存在点击一次后状态卡在 recording / processing 的问题
7. 是否在组件卸载时正确停止 recognition
8. 是否在连续点击时重复创建多个 recognition 实例

二、请把录音逻辑改成稳定的状态机

请使用明确的 voiceStatus：

voiceStatus:

* "idle"：未录音
* "requesting"：正在请求麦克风权限
* "recording"：正在聆听
* "processing"：正在处理语音
* "error"：录音失败或不可用

要求：

* idle 状态点击麦克风 → 开始录音
* requesting 成功后 → recording
* recording 状态点击麦克风 → 停止录音
* 停止后 → processing
* processing 后得到文字 → 填入输入框或自动发送
* 结束后回到 idle
* 出错后显示提示，并回到 idle

三、优先使用 Web Speech API，但必须有 fallback

请优先使用浏览器 Web Speech API：

const SpeechRecognition =
window.SpeechRecognition || window.webkitSpeechRecognition

如果支持：

* lang 设置为 "zh-CN"
* interimResults 可以为 true
* continuous 设置为 false
* maxAlternatives 设置为 1

如果不支持，或者启动失败，请不要报错，不要卡死。
请使用 fallback mock transcription：

“我梦到自己一直在赶路，但怎么都赶不上。”

也就是说，即使浏览器不支持语音识别，Demo 也必须能跑通。

四、增加明确的错误提示

请增加轻量 toast 或底部提示：

如果浏览器不支持：
“当前浏览器暂不支持语音识别，已为你生成一段示例梦境。”

如果麦克风权限被拒绝：
“没有获得麦克风权限，你也可以直接输入梦境。”

如果录音没有识别到内容：
“我没有听清楚，你可以再说一次，或者先用示例梦境继续。”

如果录音成功：
“已经听见了你的梦。”

提示不要太大，不要破坏页面美感。

五、录音按钮视觉状态

请优化麦克风按钮状态：

idle：

* 普通发光状态
* 文案可以是：“点击说梦”

requesting：

* 按钮轻微变亮
* 文案：“正在请求麦克风权限…”

recording：

* 按钮明显发光
* 有 pulse 动画
* 文案：“正在聆听你的梦…”
* 可以显示一个小的动态声波

processing：

* 按钮稍微变暗
* 文案：“正在整理你的梦…”

error：

* 短暂提示后恢复 idle

六、停止录音后怎么处理文字

请按这个逻辑处理：

1. 如果识别到了文字 transcript：

   * 把 transcript 放入输入框
   * 然后自动发送，进入聊天消息

2. 如果没有识别到文字：

   * 使用 fallback mock transcription：
     “我梦到自己一直在赶路，但怎么都赶不上。”
   * 自动发送这段文字

这样 Demo 不会因为识别失败而断掉。

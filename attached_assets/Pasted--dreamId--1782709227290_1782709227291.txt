修复“续写”功能。请严格按以下定义实现，不要再理解为新章节、新话题或新梦境。

## 核心定义

“续写”就是编辑同一条梦境记录。

它必须满足：

* 同一个dreamId；
* 同一张梦境卡片；
* 同一份聊天messages；
* 原聊天记录在聊天页面中完整可见；
* 新对话追加在原记录末尾；
* 保存时更新原记录；
* 梦境卡片数量绝对不能增加。

不是：

* 新建梦境；
* 新建卡片；
* 新建章节；
* 创建parentDreamId；
* 复制旧梦境；
* 只把摘要交给AI后开启空白聊天。

## 正确流程

假设原梦境为：

```text
dreamId = D1
messages = [M1, M2, M3, M4]
梦境走廊卡片数量 = 1
```

点击D1的“续写”后：

```text
mode = continue
editingDreamId = D1
currentMessages = [M1, M2, M3, M4]
```

进入Dream Space时，聊天区域必须立即显示：

```text
M1
M2
M3
M4
```

用户可以在聊天区域向上、向下滚动查看原来的所有消息。

用户继续产生：

```text
M5
M6
```

页面中的消息应变成：

```text
[M1, M2, M3, M4, M5, M6]
```

点击保存后：

```text
PATCH /dreams/D1
```

将M5、M6追加到D1原来的messages中。

保存完成后：

```text
dreamId仍为D1
梦境卡片数量仍为1
```

重新打开D1时，必须看到M1到M6的完整连续记录。

## 进入续写页面

点击“续写”时必须携带原dreamId，建议使用明确且不会因页面刷新丢失的方式，例如：

```text
/dream-space?continueDreamId=D1
```

或者等效的持久状态。

不要只依赖临时React state，因为页面跳转或刷新后可能丢失editingDreamId。

进入页面后：

1. 根据continueDreamId读取原梦境；
2. 将原梦境的完整messages放入聊天页面当前消息状态；
3. 恢复原人格；
4. 恢复原音乐快照；
5. 新消息追加到原messages末尾；
6. 页面不能显示为空白新聊天。

## 保存逻辑必须明确分支

新建聊天：

```text
mode = new
editingDreamId = null
```

保存时才允许：

```text
POST /dreams
```

续写聊天：

```text
mode = continue
editingDreamId = D1
```

保存时只能：

```text
PATCH /dreams/D1
```

在continue模式下，代码中禁止调用任何创建梦境的POST接口。

如果editingDreamId丢失，不要自动降级成新建卡片，应提示保存失败并保留当前聊天内容。

## 更新同一条记录

PATCH更新内容包括：

* 原messages + 本次新增messages；
* 更新后的摘要；
* updatedAt；
* 当前音乐快照；
* 当前人格。

必须保留：

* 原dreamId；
* 原createdAt；
* 原卡片；
* 原标题和封面，除非用户主动修改。

不要创建：

* parentDreamId；
* chapter；
* childDream；
* duplicateDream；
* 新卡片。

## UI要求

续写进入聊天页面后：

* 旧聊天内容完整显示；
* 可以向上滚动看以前的消息；
* 新消息自然接在最后一条旧消息后；
* 可在旧记录和新记录之间显示一条轻提示：
  “从这里继续”
* 这条提示只用于视觉分隔，不代表新章节或新记录。

## 排查当前错误

请检查：

1. 续写跳转时dreamId是否丢失；
2. currentMessages是否被初始化为空数组；
3. 保存按钮是否始终调用createDream；
4. mode是否在页面刷新后恢复成new；
5. PATCH接口是否真的更新原dreamId；
6. 保存成功后列表是否错误地append新卡片，而不是替换原卡片；
7. React Query缓存是否使用新增逻辑，而不是更新原记录。

保存成功后，梦境列表应按dreamId替换原数据：

```text
oldDream.id === updatedDream.id
```

不能把updatedDream追加到数组末尾。
